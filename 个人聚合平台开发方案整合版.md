## 一、项目概述

### 1.1 项目背景

作为山东大学（威海）翻译学院大一本科生，当前同时维护 OutEye 系列四个 AI 产品（OutEye 1.0 至 4.0），参与挑战杯揭榜挂帅（国家级，进行中）等多项学术与竞赛事务。日常面临以下核心痛点：

| 痛点 | 具体表现 | 平台解决方式 |
| ---- | -------- | ------------ |
| 多项目并行管理混乱 | OutEye 2.0/3.0/4.0 同时推进，任务散落在各处 | 多项目切换器 + 跨项目任务聚合看板 |
| 多 AI 平台切换成本高 | DeepSeek / Kimi / Claude / MiMo 间反复复制粘贴 | AI Hub 多模型并行对话 + 转发审核 |
| 个人成果展示分散 | 项目截图、证书、获奖记录无统一阵地 | 公开作品集（大图 + Mockup + 时间线） |
| AI 额度管理盲区 | MiMo 800亿 Token 缺乏监控，其他平台费用分散 | AI 额度监控台 |
| 竞赛节点跟踪缺失 | 挑战杯材料提交、答辩倒计时无显性化工具 | 竞赛指挥中心 |

### 1.2 双重定位

```
┌─────────────────────────────────────────────────────────────┐
│                    hub.outeye.cn                            │
├─────────────────────────┬───────────────────────────────────┤
│      公开区域           │          私密区域（需登录）        │
│    （个人名片）         │          （效率工作台）            │
├─────────────────────────┼───────────────────────────────────┤
│ • 首屏身份认知+形象照   │ • 竞赛指挥中心（挑战杯倒计时）     │
│ • 项目大图展示+Mockup   │ • 多项目切换器（OutEye 1-4）       │
│ • 荣誉时间线+证书预览   │ • 任务看板（用户自管理 CRUD）      │
│ • 技能图谱+工具图标     │ • AI Hub（多模型并行+转发审核）    │
│ • 联系方式              │ • 数据中心（Token/项目数据）         │
│                         │ • AI 额度监控台（MiMo/DS/Kimi）    │
│                         │ • 灵感收集                          │
└─────────────────────────┴───────────────────────────────────┘
```

### 1.3 域名与部署策略

| 项目         | 具体配置                                                            |
| ---------- | --------------------------------------------------------------- |
| **主域名**    | 现有阿里云域名（如 `outeye.top）                                          |
| **二级域名**   | `hub.outeye.top（推荐）                                             |
| **DNS 解析** | 阿里云域名解析添加 CNAME 记录 → 指向 Vercel 部署域名（如 `zhaoyan-hub.vercel.app`） |
| **SSL 证书** | Vercel 自动提供 HTTPS，无需额外配置                                        |
| **部署平台**   | Vercel（与 OutEye 系列一致，免费额度充足）                                    |

**品牌逻辑：** OutEye 是产品矩阵（外眼系列），Hub 是个人中枢（聚合个人所有项目与数据），形成品牌家族化。

## 二、需求分析

### 2.1 用户画像与展示策略

| 受众 | 核心关注点 | 平台展示策略 |
| ---- | ---------- | ------------ |
| **竞赛评委** | 技术深度、产品化能力、项目完整性 | 项目 Mockup 大图、实时数据、技术架构图、竞赛指挥中心显性化 |
| **HR / 招聘方** | 工程能力、技术栈匹配、开发活跃度 | 技能图标墙、GitHub 绿墙式提交记录、项目在线演示链接 |
| **导师 / 老师** | 学术潜力、研究态度、团队协作 | 荣誉时间线（含证书）、挑战杯进度、学术活动 |
| **投资人** | 产品矩阵、数据验证、持续迭代能力 | OutEye 系列数据总览、Token 消耗规模、多项目并行管理 |
| **个人日常** | 效率、专注、不遗漏节点 | 竞赛倒计时、跨项目任务聚合、AI 额度预警、快捷入口 |

#### 公开区域（无需登录）

| 功能模块 | 优先级 | 说明 |

| 荣誉奖项 | P1 | 纵向时间线 + 证书/现场照片 + 进行中项目进度条 + 分类筛选 |
| 技能图谱 | P1 | 图标墙（SVG）+ 熟练度进度条 + 悬停使用场景提示 |
| 联系方式 | P0 | 邮箱、微信、GitHub、LinkedIn + 联系表单（Resend API） |

#### 私密区域（需登录）

| 功能模块 | 优先级 | 说明 |
| -------- | ------ | ---- |
| 工作台首页 | P0 | 竞赛指挥中心 + 多项目切换器 + 跨项目任务聚合（CRUD） + AI 快捷入口 |
| AI Hub | P0 | 多话题 × 多窗格 × 独立上下文 × 转发审核（非旧版对比模式） |
| 数据中心 | P1 | 项目数据统计 + Token 消耗趋势 + 智能洞察预警 + GitHub 最近提交（可选卡片） |
| AI 额度监控 | P1 | MiMo 800亿进度条 + DeepSeek/Kimi/Claude 费用手动记录 + 消耗优化建议 |
| 灵感收集 | P2 | 全局快捷键 `Ctrl+Shift+I` 呼出浮层 + 后台保存 + 标签分类 |

### 2.3 非功能需求

| 需求类型 | 具体要求 |
| -------- | -------- |
| **性能** | 首屏加载 < 2s（Lighthouse 性能分 > 90），图片懒加载 + WebP |
| **安全** | NextAuth.js 认证 + HTTP-only Cookie + 中间件路由保护 + 密码存环境变量 + API Key AES 加密存储 |
| **可用性** | 响应式（Mobile / Tablet / Desktop），支持 PWA，自定义光标桌面端启用 |
| **可维护性** | 模块化架构，tRPC 端到端类型安全，便于后续扩展新模块 |
| **无障碍** | 键盘导航（Tab）、Skip Link、语义化 HTML、ARIA 标签、颜色对比度 > 4.5:1 |

## 三、前端 UI 设计系统

### 3.1 设计哲学

参考 **Linear（极简动效）+ Vercel（黑白精度）+ 2026 顶级作品集趋势（大图密集型、自定义光标、滚动视差）**：

| 原则 | 实现方式 |
| ---- | -------- |
| 极简主义 | 去除非必要边框，卡片使用半透明底色 + 1px 边框 |
| 大图叙事 | 项目展示使用设备 Mockup + 全屏截图画廊，荣誉页证书大图 |
| 微妙动效 | Framer Motion 页面过渡、滚动触发浮现、数字滚动动画 |
| 品牌光效 | 电光蓝-紫罗兰渐变作为全局强调色，悬停时边框发光跟随鼠标 |
| 呼吸感布局 | Bento Grid 大小不一的卡片拼图，充足留白 |

### 3.3 字体规范

| 场景 | 字体 | 字重 | 备注 |

**设计细节：**
- **自定义光标（桌面端）：** 电光蓝（#06b6d4）光环，直径 20px，悬停按钮时光环放大至 40px 并增强发光。移动端禁用。
- **形象照：** 圆形裁剪，边框使用 `linear-gradient(135deg, #06b6d4, #8b5cf6)`，2px 粗细，鼠标悬停时整体上浮 4px + 阴影扩散。
- **Bento Grid：** 四个亮点卡片不等大小（第一个项目卡片占 2 列，其余各占 1 列），图片占卡片高度 60%，底部文字叠加渐变遮罩确保可读性。
- **滚动动画：** 页面加载时 Hero 文字从下方 30px 淡入（duration 0.8s），Bento 卡片依次延迟 0.1s 浮现。

参考 MultipleChat 的 Compare 模式与 ChatHub 的多窗口并行设计，但更进一步：支持**同一话题下多模型独立对话 + 跨模型转发审核 + 多话题并行**。

#### 5.2.3 核心概念与数据模型

| 概念 | 说明 | 类比 |
| ---- | ---- | ---- |
| **话题（Thread）** | 一个项目/任务维度，如"OutEye 4.0 去重优化"、"挑战杯答辩" | 文件夹 |
| **窗格（Pane）** | 话题内的一个模型对话实例，每个窗格绑定一个已配置的模型 | 独立聊天窗口 |
| **消息（Message）** | 窗格内的对话记录，每个窗格独立存储 | 聊天记录 |
| **转发（Relay）** | 将某条消息从窗格 A 转发给窗格 B，让 B 审核/补充 | @某人 |

**关键规则：每个窗格独立上下文。** 窗格 A 的历史消息不会泄露给窗格 B，从根本上避免 Cherry Studio 等多模型工具出现的"上下文紊乱"问题。

#### 5.2.4 三种工作模式

**模式一：同一话题，多窗格并行（处理同一问题的不同方面）**

```
场景：优化 OutEye 4.0 评论去重逻辑

DeepSeek 窗格：    Claude 窗格：       Kimi 窗格：
"怎么优化去重？"   "帮我写单元测试"    "检查事实准确性"
        ↓                ↓                  ↓
DS 给方案         Claude 写测试        Kimi 核查数据
        ↓                ↓                  ↓
[转发给 Claude]   [转发给 Kimi]        [转发给 DS]
"请基于这个方案    "请为以下代码        "请验证这个
写代码"           生成测试用例"        方案的可行性"
```

**模式二：跨模型接力审核（形成合力）**

```
步骤：
1. 让 DeepSeek 先写一版技术方案
2. 选中 DeepSeek 的回复 → 点击"让 Claude 审核"
3. Claude 窗格自动收到："请审核以下技术方案，找出遗漏和错误：[DeepSeek 的方案]"
4. Claude 给出审核意见（发现3处问题）
5. 选中 Claude 的审核意见 → 点击"让 DeepSeek 修正"
6. DeepSeek 窗格收到："请根据以下审核意见修正原方案：[Claude 的意见]"
7. 循环直到满意
8. 最终点击"采纳为结论" → 生成最终 Markdown 导出
```

**模式三：多话题并行（同时处理不同项目）**

```
左侧话题列表：
📁 OutEye 4.0（进行中）
   └─ 窗格：DeepSeek + Claude
📁 挑战杯答辩（进行中）
   └─ 窗格：DeepSeek + Kimi
📁 Hub 开发（进行中）
   └─ 窗格：DeepSeek

点击"挑战杯答辩" → 右侧切换为该话题的窗格布局
各话题完全隔离，互不干扰
```

#### 5.2.5 底部输入栏交互逻辑

```
┌─────────────────────────────────────────────────────────────────┐
│  [输入消息...                                                   │
│   支持 Markdown 快捷输入]                                       │
│                                                                 │
│  @选择模型  [DeepSeek ✓] [Claude ✓] [Kimi ○] [MiMo ○]          │
│                                                                 │
│  [发送模式 ▾]  [发送]                                           │
│                                                                 │
│  发送模式：                                                     │
│  ├─ 发送给已选中的模型（默认）→ 所有打勾的窗格同时收到消息      │
│  ├─ 仅发送给当前激活窗格 → 只发给鼠标最后点击的窗格             │
```

**转发功能（消息级别）：**

每条 AI 回复下方有操作栏：
- `✎ 继续问`：在该窗格继续追问
- `↗️ 转发给...`：弹出选择器，选择另一个模型窗格 → 该窗格收到"请审核/补充以下内容：[原消息]"
- `📋 复制`：复制到剪贴板
- `⭐ 收藏`：收藏到灵感库

#### 5.2.6 上下文与记忆管理

**上下文隔离策略（解决多模型对话的核心技术难点）：**

| 层级 | 隔离方式 | 实现 |
| ---- | -------- | ---- |
| **话题间隔离** | 不同话题的消息存储在不同 `thread_id` 下 | 数据库层面隔离 |
| **窗格间隔离** | 同一话题下不同窗格的消息存储在不同 `pane_id` 下 | 数据库层面隔离 |
| **模型间隔离** | 每个窗格调用 API 时只发送该窗格的 `messages` 数组 | API 调用层面隔离 |

**记忆恢复：**
- 打开话题时，自动从 Supabase 加载该话题下所有窗格的历史消息
- 每个窗格渲染为 `[{role: 'user', content: '...'}, {role: 'assistant', content: '...'}, ...]`
- 发送新消息时，将该窗格的全部历史消息 + 新消息一起发送给对应模型 API
- 支持"清空上下文"按钮（仅清空该窗格，不影响其他窗格）

**上下文长度管理：**
- 当某窗格消息总 Token 超过模型上限的 80% 时，提示"上下文即将满载"
- 提供"总结并压缩上下文"功能：调用 DeepSeek 将该窗格历史对话总结为 500 字摘要，替换为系统提示词

#### 5.2.7 与 Data Center 的联动

- 每次 API 调用自动记录 Token 消耗到 `token_usage` 表
- 在 AI Hub 右下角显示本次话题的累计消耗
- 支持"导出对话记录"为 Markdown（含所有窗格的完整对话）

```typescript
      name: 'Admin Access',

      },

  callbacks: {
    async jwt({ token, user }) { if (user) token.role = user.role; return token },
```

### 6.3 时区处理策略（关键修正）

**问题：** Vercel 服务器默认 UTC 时区，"今天"的判断极易出现 8 小时偏差。

**解决方案：**
- 数据库统一存储 UTC 时间（`TIMESTAMPTZ`）。
- 前端展示时，强制使用 `dayjs` 转换为 `Asia/Shanghai` 时区。
- 所有"今日任务"、"截止日期"计算均在客户端完成。

```typescript
import dayjs from 'dayjs'
import utc from 'dayjs/plugin/utc'
import timezone from 'dayjs/plugin/timezone'
dayjs.extend(utc); dayjs.extend(timezone)
const displayTime = dayjs(utcTime).tz('Asia/Shanghai').format('YYYY-MM-DD HH:mm')
```

### 6.5 安全设计

- **环境隔离：** 开发/生产使用不同的 `ai_models` 配置（通过环境变量区分）

## 附录 B：决策理由与思路

### B.1 为什么选择 Next.js 14 App Router？

- **SSR + SSG 混合渲染：** 公开区域使用 SSG 预渲染，私密区域使用 SSR 动态渲染，兼顾性能与实时性。
- **tRPC 深度集成：** 端到端类型安全，从数据库到前端无需手动定义 API 类型。
- **Vercel 原生部署：** 零配置部署，自动 CI/CD，与现有 OutEye 系列技术栈一致。
- **生态成熟：** shadcn/ui、Framer Motion、NextAuth.js 等库高度适配。

### B.2 为什么取消内嵌在线演示？

- **iframe 跨域问题：** OutEye 2.0/3.0/4.0 分别部署在不同子域，iframe 需处理 CSP、Cookie 共享等问题。
- **移动端适配：** iframe 在移动端的缩放、滚动体验极差。
- **维护成本：** 每次 OutEye 系列更新，Hub 需同步验证 iframe 嵌入是否正常。
- **替代方案更优：** 设备 Mockup + 截图画廊 + 外部链接，视觉更专业，维护成本更低。

### B.3 为什么删除开发运维中心？

- **手动记录负担重：** Claude Code 会话追踪需手动填写耗时、修改内容，用户反馈"与提质增效目标相悖"。
- **功能重复：** GitHub 提交记录可作为数据中心的可选卡片，无需独立成页。
- **降低 MVP 复杂度：** Phase 1 专注核心功能，运维追踪可后续按需添加。

### B.4 为什么重构 AI 协作中心为 AI Hub？

- **原设计不实用：** "统一输入→分栏对比→差异高亮"的对比工作流，实际使用中发现：
  - 各模型回答长度、格式差异大，对比困难
  - 自动差异高亮准确度有限，反而增加认知负担
  - 无法实现"接力审核"，各模型独立回答，未形成合力
- **新设计更灵活：** 多话题 × 多窗格 × 独立上下文 × 转发审核，支持：
  - 同一问题让不同模型从不同角度分析
  - A 模型的回答转发给 B 模型审核/补充
  - 真正形成"AI 团队"协作效果

### B.5 为什么任务采用自管理 CRUD？

- **避免过度自动化：** 自动抓取需要对接多个外部系统（GitHub Issues、日历等），实现复杂且不可靠。
- **用户完全掌控：** 自定义添加、编辑、删除、排序，灵活度最高。
- **降低开发成本：** CRUD 是最成熟的数据管理模式，开发速度快，bug 少。
